# 07. 可视化、报告与交付规范

## 1. 可视化总体原则

图表必须服务于结论。

必须遵守：

- 图表类型与数据关系匹配。
- 标题简洁明确。
- 坐标轴、单位、时间范围清晰。
- 分类过多时进行合理排序或合并。
- 颜色使用有含义，避免过度装饰。
- 不使用误导性坐标轴。
- 不隐藏关键异常值，除非明确说明。

禁止输出没有解释、没有业务含义、没有后续使用价值的图表。

## 2. 图表信息要求

每张关键图表必须包含：

- 图表标题。
- 横轴和纵轴含义。
- 单位。
- 时间范围。
- 数据口径。
- 必要的图注。

如果图表经过抽样、过滤、截尾、标准化或聚合，必须在 Notebook 中说明。

## 3. 图表导出要求

关键图表必须导出到：

```text
reports/figures/
```

建议图表命名格式：

```text
fig_01_metric_trend.png
fig_02_segment_comparison.png
fig_03_model_performance.png
```

Notebook 中展示的关键图表，必须能在 `reports/figures/` 中找到对应文件。

## 4. 结果表导出要求

关键结果表必须导出到：

```text
reports/tables/
```

常见结果表包括：

- 数据质量检查汇总。
- 核心指标汇总。
- 分组对比结果。
- 统计检验结果。
- 模型评估结果。
- 关键样本清单，若权限允许。

结果表必须字段清晰、命名明确，并避免包含未脱敏敏感信息。

## 5. 报告结构与交付规范

最终交付必须包含脚本生成的结果呈现型数据分析报告，而不是只交付图表、结果表或 Notebook 输出。

正式分析前必须先确认最终报告结构：

```text
reports/final/final_report_structure.md
```

报告结构模板为：

```text
docs/final_report_structure_template.md
```

最终报告结构必须把汇总结果放在最前面，并说明每个章节：

- 本节目标。
- 具体分析逻辑。
- 预期呈现的关键结果。
- 预期图表。
- 预期结果表。
- 口径与限制。

最终报告由脚本生成：

```bash
python scripts/generate_final_report.py
```

脚本默认读取：

```text
reports/final/report_inputs.yaml
```

并生成：

```text
reports/final/final_analysis_report.md
```

默认报告只呈现结果、图表、表格、口径和输出文件路径，不自动生成洞察、归因、业务建议或扩写解读。

如需渲染已准备好的洞察内容，必须由用户明确要求，并显式运行：

```bash
python scripts/generate_final_report.py --with-insights
```

最终交付文件应该保存到：

```text
reports/final/
```

交付内容必须能够回答：

- 本次分析服务什么业务决策？
- 业务问题是如何拆解的？
- 汇总结果是什么？
- 每个业务问题对应的结果是什么？
- 对应图表和结果表在哪里？
- 口径、限制和输出文件是什么？

禁止只交付代码或 Notebook，而不提供面向业务理解的结论。

禁止绕过 `scripts/generate_final_report.py` 手工拼接最终报告，除非用户明确要求。

禁止默认生成大模型洞察或业务解读。只有用户明确要求时，才可以渲染或加入洞察内容。

## 6. 结果与可选洞察表达规范

默认最终报告只呈现结果，不自动生成洞察、归因、业务建议或扩写解读。

如果用户明确要求加入洞察或业务解读，则洞察内容必须区分：

- 事实。
- 推断。
- 建议。
- 局限性。

可选洞察建议采用以下表达结构：

```markdown
## 核心发现

1. 发现一：说明具体数据事实、影响范围和业务含义。
2. 发现二：说明分组差异、趋势变化或异常现象。
3. 发现三：说明需要关注的风险或机会。

## 业务建议

1. 建议一：说明建议动作、目标对象和预期影响。
2. 建议二：说明执行优先级和验证方式。

## 风险与局限性

1. 数据局限：说明数据覆盖、质量或口径问题。
2. 方法局限：说明统计或模型方法的假设限制。
3. 业务局限：说明结论落地时需要额外验证的条件。

## 下一步

1. 需要补充的数据。
2. 需要验证的假设。
3. 可以开展的实验或监控。
```

## 7. 业务建议要求

默认最终报告不自动生成业务建议。

如果用户明确要求加入业务建议，则业务建议必须可执行。

建议应该说明：

- 建议动作。
- 目标对象。
- 预期影响。
- 执行优先级。
- 验证方式。
- 风险或前置条件。

禁止输出空泛建议，例如“继续优化”“加强运营”“提升转化”，除非说明具体动作和验证方法。

## 8. 局限性说明要求

最终报告必须说明局限性。

常见局限性包括：

- 数据覆盖不足。
- 数据质量问题。
- 样本量不足。
- 指标口径限制。
- 时间窗口限制。
- 方法假设限制。
- 无法支持因果解释。
- 结果只适用于特定人群或场景。

局限性不是附带免责声明，而是帮助读者正确使用结论的必要信息。
