# 05. 数据管理、质量检查与清洗规范

## 1. 数据分层

数据必须按处理阶段分层保存：

- `data/raw/`：原始数据，禁止修改。
- `data/interim/`：清洗或转换过程中的中间数据。
- `data/processed/`：最终用于分析、建模或报告的数据。
- `data/external/`：外部数据源。

如果数据来自数据库且不落地保存，Notebook 必须说明原因，并记录 SQL 文件和查询参数。

## 2. 原始数据保护

禁止修改 `data/raw/` 中的原始数据文件。

如需修正原始数据问题，必须：

- 在清洗逻辑中实现。
- 在 Notebook 中说明。
- 输出到 `data/interim/` 或 `data/processed/`。
- 保留修正规则。

## 3. 敏感数据处理

如果数据包含个人信息、交易信息、联系方式、证件信息、精确地址或其他敏感字段，必须：

- 在项目说明中标注敏感字段类型。
- 只读取分析所需字段。
- 尽可能脱敏或聚合。
- 禁止在报告、图表、日志和 Notebook 输出中暴露明细敏感信息。
- 禁止提交真实敏感数据到版本库。

## 4. 数据源与字段理解

在清洗数据前，必须先建立数据画像。

必须说明：

- 数据来源。
- 数据表或文件名称。
- 字段含义。
- 数据粒度。
- 时间范围。
- 主键或业务唯一键。
- 外键或表间关系。
- 数据更新频率。
- 数据可信度。

建议输出数据字典，保存到：

```text
reports/tables/data_dictionary.csv
```

## 5. 数据质量检查范围

每个项目必须进行数据质量检查。

至少检查：

- 行数和列数。
- 字段类型。
- 主键或业务唯一键。
- 重复记录。
- 缺失值。
- 异常值。
- 时间范围。
- 取值范围。
- 分类字段枚举值。
- 关键指标汇总值。
- 多表 join 后的行数变化。

数据质量检查结果必须在 Notebook 中展示，并可以导出到：

```text
reports/tables/data_quality_summary.csv
```

## 6. 数据质量问题记录

如发现数据质量问题，必须说明：

- 问题字段。
- 问题规模。
- 影响范围。
- 处理方式。
- 是否影响最终报告结果。

禁止静默删除大量数据。任何删除、过滤、填补、截尾、标准化、去重操作，都必须说明规则和影响行数。

## 7. 多表关联检查

涉及多表 join 时，必须检查：

- join key 是否唯一。
- join 前后行数变化。
- 是否出现重复膨胀。
- 是否出现大量未匹配记录。
- 未匹配记录是否符合业务预期。
- join 类型是否合理。

如果 join 后指标发生明显变化，必须解释原因。

## 8. 数据清洗要求

数据清洗必须可追溯、可复现。

必须记录：

- 删除了哪些数据。
- 为什么删除。
- 删除前后行数变化。
- 缺失值如何处理。
- 异常值如何处理。
- 字段类型如何转换。
- 新增字段如何计算。
- 多表如何关联。
- 聚合粒度如何定义。

清洗后的关键数据应该保存到：

```text
data/processed/
```

如果因数据敏感或数据量过大不保存处理结果，必须在 Notebook 中说明替代复现方式。

## 9. 缺失值处理规范

处理缺失值前必须先分析缺失模式。

必须说明：

- 哪些字段存在缺失。
- 缺失比例。
- 缺失是否集中在特定时间、渠道、地区或用户群。
- 缺失是否有业务含义。
- 删除、填补或保留的理由。

禁止无说明地使用均值、中位数、众数或 0 填补。

## 10. 异常值处理规范

处理异常值前必须区分：

- 数据错误。
- 极端但真实的业务现象。
- 由口径不一致造成的异常。
- 由系统或采集问题造成的异常。

异常值处理必须说明：

- 异常识别规则。
- 影响记录数量。
- 处理方式。
- 是否影响核心结论。

禁止为了让图表更好看而静默删除异常值。

## 11. 数据转换规范

数据转换必须说明：

- 转换字段。
- 转换规则。
- 转换前后数据类型。
- 是否改变业务含义。
- 是否影响后续指标计算。

涉及时间字段时，必须明确：

- 时区。
- 日期截断规则。
- 自然日、自然周、自然月的定义。
- 是否包含起止边界。

## 12. 输出数据要求

保存到 `data/processed/` 的数据必须具备：

- 清晰粒度。
- 明确字段含义。
- 可追溯来源。
- 与主 Notebook 中的分析流程一致。
- 不包含未脱敏敏感信息，除非项目权限明确允许。

建议同时导出处理说明或数据字典，便于后续复查。
